Q-1(a) Illustrate and explain any one embedded system with a necessary diagram. 5 Marks · A, U · CO1

Example: Automatic Temperature Control System

An embedded system is a dedicated computing system designed to perform one or a few specific functions, often with real-time constraints, embedded within a larger device.

System: Room Temperature Controller

This system reads temperature using a sensor, displays it on an LCD, and controls a fan/heater based on set thresholds — all managed by a microcontroller.

┌──────────────────────────────────────────────────┐ │ TEMPERATURE CONTROL SYSTEM │ │ │ │ [Temperature]──ADC──┐ │ │ Sensor (LM35) │ │ │ ▼ │ │ ┌───────────────┐ ┌─────────┐ │ │ [Keypad]───▶│ MSP430 / AVR │───▶│ 16×2 │ │ │ (Set Temp) │ Microcontroller│ │ LCD │ │ │ │ │ └─────────┘ │ │ [Power]────▶│ GPIO / PWM │───▶[FAN / HEATER]│ │ Supply └───────────────┘ (Relay / MOSFET│ │ │ │ │ [Buzzer] │ │ (Alarm if >Max) │ └──────────────────────────────────────────────────┘

Component Description

ComponentFunction
LM35 Temperature SensorOutputs analog voltage proportional to temperature (10 mV/°C)
ADC (built-in)Converts analog sensor output to digital value
MicrocontrollerCore unit — reads ADC, processes data, drives outputs
16×2 LCDDisplays current and set temperature
Relay / MOSFETSwitches fan or heater based on microcontroller decision
KeypadUser input to set desired temperature threshold
BuzzerAlarm if temperature exceeds safe limit

Working Principle

  1. LM35 outputs voltage = 10 mV × Temperature (°C). At 25°C → 250 mV.
  2. ADC converts this to a 10-bit digital value; microcontroller computes actual °C.
  3. If temperature > set threshold → turn ON fan (cooling) via relay.
  4. If temperature < set threshold → turn ON heater.
  5. LCD continuously displays both the measured and set values.
  6. If temperature exceeds a safety maximum → buzzer sounds an alarm.
Key Properties: This system has dedicated functionality (only controls temperature), operates in real-time (immediate response), is embedded in a device, consumes low power, and runs without user OS intervention.
Q-1(b) Analyze the multiplexing scheme in the MSP430 processor for the port pins and explain its functionality. 4 Marks · N · CO1

Port Pin Multiplexing in MSP430

MSP430 has limited physical pins. To maximize utility, most I/O pins are multiplexed — each pin can serve multiple functions selectable via software registers.

Concept of Multiplexing

Each physical pin can function as: (1) General Purpose I/O (GPIO), or (2) a peripheral function like UART TX/RX, SPI MOSI/MISO/CLK, Timer output, ADC input, I2C SDA/SCL, etc. Software selects which function is active through control registers.

Example (C Code)

// Configure P1.1 as UART RXD, P1.2 as UART TXD P1SEL |= BIT1 + BIT2; // Select peripheral function P1SEL2 |= BIT1 + BIT2; // Select USCI_A (UART) // Configure P1.0 as GPIO output (LED) P1DIR |= BIT0; // PxSEL = 0 (default) → GPIO P1OUT |= BIT0; // Set output HIGH
Advantage of Multiplexing: Reduces chip size and pin count while offering maximum flexibility. The designer selects only the peripherals needed for their specific application.

Evaluation of Scheduling Algorithms

AlgorithmPrincipleStrengthsLimitationsBest For
Round Robin (RR)Each task gets equal CPU time slice in rotationFair, simple, no starvationIgnores priority; hard to meet deadlinesSoft real-time, background tasks
RR with InterruptRR base + high-priority interrupts preempt RR tasksHandles urgent events quicklyInterrupt overload can starve RR tasksMixed-priority systems

Earliest Deadline First (EDF)Task with nearest absolute deadline runs first (dynamic priority)Theoretically optimal (100% utilisation)Complex to implement; can miss multiple deadlines under overloadDynamic hard real-time

For simple embedded systems, function queue scheduling or RR with interrupts is sufficient. Safety-critical systems require fault-tolerant scheduling to guarantee operation under hardware failure.
Q-4 Explain the SPI protocol and analyze its working mechanism in embedded systems with necessary diagram. 7 Marks · U, N · CO2

SPI – Serial Peripheral Interface

SPI is a synchronous, full-duplex, 4-wire serial communication protocol developed by Motorola. It uses a master-slave architecture where one master controls multiple slaves using a shared bus plus individual chip-select lines.

SPI Signal Lines

SignalFull NameDirectionFunction
SCLKSerial ClockMaster → SlaveClock signal generated by master; synchronises all data transfers
MOSIMaster Out Slave InMaster → SlaveData line from master to slave
MISOMaster In Slave OutSlave → MasterData line from slave to master
CS / SSChip Select / Slave SelectMaster → SlaveActive LOW — selects which slave is active

SPI Block Diagram

MASTER SLAVE ┌─────────────────┐ ┌─────────────────┐ │ ┌───────────┐ │ SCLK ──────│────────────── │ │ │ 8-bit │ │ │ 8-bit │ │ │ Shift │◀─┼── MISO ───────│── Shift │ │ │ Register │ │ │ Register │ │ │ │──┼── MOSI ──────▶│ │ │ └───────────┘ │ └─────────────────┘ │ Clock Gen ──┼── SCLK ──────▶ │ │ │ CS ──────▶ (Chip Select) │ └─────────────────┘ Note: Each SCLK pulse shifts 1 bit out of master's register and 1 bit into master from slave (simultaneously). After 8 clock pulses → complete 1-byte exchange.

SPI Working Mechanism

  1. Master pulls CS LOW to select the target slave.
  2. Master generates SCLK pulses. Each pulse shifts 1 bit: MOSI sends master's data to slave, MISO brings slave's data to master.
  3. Both master and slave shift registers rotate simultaneously — after 8 pulses, a full byte is exchanged in both directions (full-duplex).
  4. Master pulls CS HIGH to deselect the slave and end the transaction.
  5. For multiple slaves, each has its own CS line; others are kept HIGH (inactive).

I2C – Inter-Integrated Circuit Protocol

I2C (pronounced "I-squared-C") is a synchronous, half-duplex, 2-wire serial communication protocol developed by Philips (now NXP). It supports multiple masters and multiple slaves on the same 2-wire bus using a unique 7-bit address for each device.

SDA (Serial Data)Bidirectional data line — carries address and data bitsOpen-drain with pull-up resistor
SCL (Serial Clock)Clock line generated by masterOpen-drain with pull-up resistor

I2C Working Mechanism — Step by Step

  1. START Condition: Master pulls SDA LOW while SCL is still HIGH. This signals the beginning of communication to all slaves on the bus.
  2. Address + R/W bit: Master sends 7-bit slave address (MSB first) + 1 R/W bit (0=write, 1=read). Total 8 bits per address byte.
  3. ACK from Slave: The addressed slave pulls SDA LOW during the 9th clock pulse. If SDA remains HIGH, it's a NACK (not acknowledged — device not present or busy).
  4. Data Transfer: Each byte (8 bits) is sent MSB first on SDA, clocked by SCL. After each byte, an ACK/NACK is sent.
  5. Repeated START (Sr): Master can send a new START without releasing bus — used to switch from write to read within one transaction.
  6. STOP Condition: Master releases SDA (HIGH) while SCL is HIGH, ending the transaction and freeing the bus.

I2C Read Sequence

Master: START → [Addr + W] → [Reg Address] → Sr → [Addr + R] → ...NACK → STOP Slave: ACK ACK ACK → [Data] ... Example: Read register 0x00 from I2C device at address 0x48: 1. START 2. Send 0x90 (0x48 << 1 | 0 = Write mode) → ACK 3. Send 0x00 (register pointer) → ACK 4. Repeated START 5. Send 0x91 (0x48 << 1 | 1 = Read mode) → ACK 6. Receive data byte from slave → NACK (last byte) 7. STOP

Applications: EEPROM (AT24C256), temperature sensors (LM75, DS3231 RTC), accelerometers (MPU6050), OLED displays (SSD1306), battery gas gauges. I2C is preferred when many devices share a bus and wiring simplicity matters more than raw speed.
Q-5 Analyze and discuss the low-power features of MSP430 processor. 7 Marks · N · CO1

Low-Power Features of MSP430

Low power consumption is the defining design philosophy of MSP430. Every architectural decision — from voltage levels to clock architecture to peripheral design — is driven by minimising energy consumption.

1. Low Supply Voltage

MSP430 operates from 1.8V to 3.6V. Dynamic power consumption follows P = CV²f. At 1.8V vs 5V (8051), the power reduction is (1.8/5)² = 0.13 — only 13% of the power at same frequency. This allows operation from a single coin cell (CR2032, 3V) for years.

2. Flexible Clock Architecture

Clock SourceFrequencyCurrentDescription
LFXT (32.768 kHz crystal)32.768 kHz~0.5 µAUltra-low-power timekeeping; active in LPM3
DCO (Internal RC)100 kHz–16 MHz~100 µAFast startup (<6 µs); no external component
VLO (Internal)~12 kHz~0.5 µAUltra-low power; alternative when crystal not needed
HFXT (External)1–16 MHz~200 µAHigh-accuracy; for precise communication

Three derived clocks: MCLK (CPU), SMCLK (fast peripherals), ACLK (slow peripherals). Each can be independently gated.

3. Five Low Power Modes (LPM0–LPM4)

4. Interrupt-Driven Architecture

MSP430 is designed to sleep between events. All peripherals (Timer, UART, ADC, I2C) can generate interrupts to wake the CPU. The CPU runs only for the microseconds needed to process data, then returns to sleep — this is called the "sleep most of the time" design pattern.

Timeline: ──sleep──▶ISR(10µs)──sleep──▶ISR(10µs)──sleep──▶... Active Duty Cycle = 10µs / 1,000,000µs = 0.001% Average Current ≈ 0.001% × 200µA + 99.999% × 1µA ≈ 1.002 µA

5. Peripherals Operate Without CPU

Timer_A, RTC, WDT, and USCI all continue functioning in LPM3 while the CPU is fully off, consuming only nanoamperes themselves. ADC can be triggered by a timer and store results via DMA — without ever waking the CPU.

6. Fast Wake-up

DCO starts in under 6 µs (unlike crystals which need 1–100 ms). This means the CPU can wake, process data in a few microseconds, and return to sleep — making frequent wake-ups energy-efficient.

7. Low-Power Design Techniques (C Code)

// 1. Enter LPM3 (CPU off, ACLK timer running) __bis_SR_register(LPM3_bits + GIE); // 2. Wake up from ISR and go back to sleep #pragma vector = TIMERA0_VECTOR __interrupt void Timer_ISR(void) { do_work(); // __bic_SR_register_on_exit(LPM3_bits); // Only if need to stay awake // Without the above line, CPU returns to LPM3 after ISR } // 3. Set all unused pins to output LOW to prevent floating P1DIR = 0xFF; P1OUT = 0x00; P2DIR = 0xFF; P2OUT = 0x00;
Result: A typical MSP430 data-logger application running at 1-second intervals in LPM3 consumes ~2 µA average. A CR2032 coin cell (220 mAh) can power it for 220,000/0.002 = 110,000,000 hours — over 12,000 years theoretically (limited by battery self-discharge in practice to ~10 years).

MSP430 Timer Module (Timer_A)

Timer_A is a 16-bit general-purpose timer in MSP430. It is one of the most important and versatile peripherals — used for time delays, periodic interrupts, PWM signal generation, frequency measurement, and event capture.

Timer_A Internal Architecture

MODE 1: Stop Mode (MC = 00)

The timer counter (TAR) is frozen — it stops counting. Used to pause the timer temporarily without losing the current count value.

TAR: ──[frozen value]──────────────────────────────────

MODE 2: Up Mode (MC = 01)

Timer counts from 0 up to the value in TACCR0, then resets to 0 and repeats. Period = (TACCR0 + 1) × clock period. An interrupt is generated every time TAR reaches TACCR0 (CCIFG0 flag set).

TAR value: TACCR0 ──│ /│ /│ /│ /│ │ / │ / │ / │ / │ │ / │ / │ / │ / │ │ / │ / │ / │ / │ 0─┘/ └┘/ └┘/ └┘/ └ ↑ ↑ ↑ ↑ INT at each TACCR0 match Period = (TACCR0 + 1) × (1 / f_clock) Example: ACLK = 32768 Hz, TACCR0 = 32767 → Period = 1 second
// 1-second interrupt using Up mode + ACLK TACTL = TASSEL_1 | MC_1 | TACLR; // ACLK, Up mode, clear TACCR0 = 32768 - 1; // 1 second TACCTL0 = CCIE; // Enable CCR0 interrupt

MODE 3: Continuous Mode (MC = 10)

Timer counts from 0 to 0xFFFF (65535) then overflows to 0 and continues. Period = 65536 × clock period. Used when more flexible timing is needed, or when multiple independent time intervals are needed using CCR1 and CCR2.

TAR: 0xFFFF──│ /│ /│ │ / │ / │ │ / │ / │ 0─┘─────────/ └────────/ └─ ↑ ↑ Overflow INT Overflow INT (TAIFG)

MODE 4: Up/Down Mode (MC = 11)

Timer counts up from 0 to TACCR0, then back down to 0, creating a triangular waveform. Period = 2 × TACCR0 × clock period. This mode is ideal for centre-aligned PWM, which produces symmetric PWM with less harmonic distortion than edge-aligned (Up mode) PWM.

TAR: TACCR0──│ /\ /\ /\ │ / \ / \ / \ │ / \ / \ / \ 0─┘─/ \/ \/ \─ Period = 2 × TACCR0 × T_clock PWM duty cycle controlled by TACCR1 (set/reset points on up and down slopes)

Capture Mode

In capture mode, TAR's current value is automatically copied into a CCRx register when a selected edge (rising/falling) is detected on an external input pin. Used for frequency measurement and pulse-width measurement.

External Signal: ───────┐ ┌────── └────────┘ ↑ ↑ TAR captured TAR captured into CCR1 (t1) into CCR1 (t2) Pulse Width = (t2 - t1) × T_clock Frequency = 1 / ((t2 - t1) × T_clock)

PWM Generation using Timer_A

// 1 kHz PWM, 50% duty cycle, using SMCLK at 1 MHz TACTL = TASSEL_2 | MC_1 | TACLR; // SMCLK, Up mode TACCR0 = 1000 - 1; // Period = 1000 cycles = 1 ms = 1 kHz TACCR1 = 500; // 50% duty cycle TACCTL1 = OUTMOD_7; // Reset/Set: output HIGH until CCR1, then LOW P1DIR |= BIT2; // P1.2 as output P1SEL |= BIT2; // Connect P1.2 to Timer_A output

Summary: Timer_A is the backbone of MSP430 timing. Up mode = clean periodic interrupts. Continuous mode = flexible multi-channel timing. Up/Down mode = centre-aligned PWM. Capture mode = measure external signals. Combined with ACLK in LPM3, Timer_A enables precise timekeeping at ~1 µA total current.
